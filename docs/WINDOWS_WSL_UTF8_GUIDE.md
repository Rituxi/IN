# Windows / WSL 中文文件安全指南

本文档用于规范本项目在 Windows 和 WSL 环境下处理中文文件的方式，目标是减少乱码、误判编码、批量覆盖等风险。

---

## 1. 结论先看

- 处理中文 Markdown、批量文本修改时，优先使用 WSL。
- 不要把 PowerShell 终端里的乱码直接当成文件真的坏了。
- 批量修改前先备份，或先提交 Git。
- 先抽样改 1 到 2 个文件，确认正常后再扩大范围。

---

## 2. 推荐环境

- Windows：浏览器、界面调试、图形化操作
- WSL：Git、AI IDE、批量文本处理、批量脚本

Windows 原生 PowerShell 仅适合轻量操作，不适合直接对大量中文文件做覆盖式改写。

---

## 3. Windows 原生环境的最低安全要求

如果必须在 Windows PowerShell 中工作，先执行：

```powershell
chcp 65001
[Console]::InputEncoding = [System.Text.UTF8Encoding]::new($false)
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new($false)
$OutputEncoding = [System.Text.UTF8Encoding]::new($false)
```

说明：

- 这只能降低风险，不能保证完全避免乱码。
- 终端显示正常，也不等于文件编码一定没有问题。

---

## 4. 批量修改前的标准流程

1. 先确认本次任务是否涉及中文文本文件。
2. 如果涉及批量修改，先确认存在 Git 可回退点，或先备份项目目录。
3. 先让工具只修改 1 到 2 个样本文件。
4. 用编辑器或实际应用打开确认中文是否正常。
5. 确认没问题，再执行批量修改。

---

## 5. 出现疑似乱码时的处理规则

不要立刻做这些事：

- 不要直接全量另存为 UTF-8
- 不要批量替换中文
- 不要因为终端显示异常就重写原文件

应该先做这些事：

1. 判断是终端显示异常，还是文件内容真的损坏。
2. 用编辑器实际打开文件确认。
3. 检查最近一次批量脚本、格式化、转码、导入导出是否改写了文件。
4. 如果已经用了 Git，优先对比变更再处理。

---

## 6. 本项目已启用的保护

- `AGENTS.md`：约束 AI IDE 的执行方式
- `.editorconfig`：统一编码与换行风格
- `.gitattributes`：约束常见文本文件使用 LF

---

## 7. 一句话原则

先确认来源，再修改文件；先样本验证，再批量处理。
