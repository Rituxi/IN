# AGENTS.md

本项目给 AI IDE、Agent、自动化脚本的维护约束如下。

## AI 执行优先级

- 本文件中的中文编码与批量编辑规则，优先级高于常规格式化、重构、整理、清理类请求。
- 只要任务涉及中文文本文件、Markdown、批量替换、批量重写，默认按高风险任务处理。
- 如果 AI 无法确认“显示乱码”是否等于“文件损坏”，必须先停止写入，再验证原因。

## 编码与文本

- 所有源码、配置、文档统一使用 `UTF-8` 编码。
- 不要把 UTF-8 文件另存为 ANSI、GBK、UTF-16。
- 修改包含中文的文件前，先确认编辑器当前编码是 `UTF-8`。
- 不要把 PowerShell `Get-Content`、`type`、`cat` 的终端输出，当成中文是否正常的唯一判断依据。
- 如果发现中文已经乱码，不要在乱码原文上继续局部拼接，先确认是文件损坏还是终端显示异常。
- 不要批量执行会改变整仓库编码的脚本。

## 查找与阅读

- 优先使用 `rg` / `rg --files` 查找内容和文件。
- 本机已安装可用的快速版 ripgrep：Windows PowerShell 下优先使用 `C:\Users\Rituxi\scoop\shims\rg.exe`，WSL/Ubuntu 下优先使用 `~/.local/bin/rg`。
- 后续搜索定位时，先用上述已验证路径；不要先尝试 Codex 自带的 WindowsApps 路径。
- 如果 `rg` 命中 `C:\Program Files\WindowsApps\OpenAI.Codex\...\rg(.exe)` 这类 Codex 自带路径，或已经出现 `Access is denied` / `Permission denied`，默认视为当前环境不可用，不要在同一任务里反复重试。
- 这种情况下，优先使用 WSL 中单独安装的原生 `rg` 或其他原生搜索工具；如果 WSL 里没有可用的原生 `rg`，再退回 PowerShell 的 `Get-ChildItem` 和 `Select-String`。
- 如果当前环境不能正常使用 `rg`，再退回 PowerShell 的 `Get-ChildItem` 和 `Select-String`。
- 先读相关文件，再改代码，不要凭猜测直接重写接口或数据结构。
- 如果只是定位内容，可以使用搜索工具；如果要判断中文编码是否正常，必须采用比终端直出更可靠的方式验证。

## 编辑方式

- 小改动优先精确修改，避免无关格式化。
- 大文件如果已经出现乱码，先验证是否真是文件损坏，再决定是否按模块整段重写。
- 不要随意改动 API 路径、字段名、Redis key、环境变量名。
- 修改 UI 时，优先抽出统一的颜色、按钮、表格样式，避免每页各写一套。
- 涉及中文 `.md`、`.json`、`.js`、`.ts`、`.tsx`、`.css`、`.html` 文件时，先小范围修改 1 到 2 个文件验证结果，再决定是否批量处理。
- 未经用户明确同意，不要对中文内容执行“全仓整理”“批量格式化”“批量重写”。
- 只要任务涉及批量编辑文本文件，开始前必须确认 Git 可回退，或先提醒用户备份。

## 中文与文案

- 中文文案要求简洁、自然、面向普通用户。
- 不要引入半截乱码、问号乱码、混乱标点。
- 新增中文时，优先使用简体中文，保持术语前后一致。

## Windows / WSL 规则

- 优先在 WSL 环境中处理中文文本和批量文件操作。
- 如果仍在 Windows 原生 PowerShell 中工作，应默认提高风险等级。
- 如果用户已经安装 WSL，涉及批量文本处理、Markdown 重写、脚本批量改写时，优先建议在 WSL 中执行。

## AI 禁止行为

- 禁止仅凭终端输出乱码，就认定原文件编码错误。
- 禁止为了“统一格式”而重写整个中文文件。
- 禁止在未抽样验证前，对中文文件执行批量脚本修改。
- 禁止在 Windows 原生 PowerShell 中直接对大量中文文件做批量覆盖式改写。
- 禁止把“可能是终端乱码”的问题，误处理成“文件需要转码”。

## AI 必做检查

- 改中文文件前，先判断当前环境是 WSL 还是 Windows PowerShell。
- 批量改动前，先检查是否存在 Git 可回退点或备份。
- 涉及多文件文本任务时，先从 1 到 2 个样本文件开始。
- 写入后，优先用编辑器实际显示、应用实际显示或显式编码读取方式复核。

## 安全边界

- 不要执行会覆盖用户现有改动的危险命令。
- 不要使用 `git reset --hard`、`git checkout --` 之类的破坏性命令，除非用户明确要求。
- 后台相关改动不能破坏现有登录、用户管理、兑换码、配额配置功能。

## 提交前检查

- 至少执行一次类型检查或构建检查。
- 检查主要页面中文是否正常显示。
- 检查按钮、表格、输入框是否还能正常交互。

## 标准工作流程

- 第 1 步：先确认是否存在中文文本文件，以及是否要批量修改。
- 第 2 步：如果要批量修改，先确认 Git 可回退，或先提醒用户备份。
- 第 3 步：优先使用 WSL；如果不能使用 WSL，再在 Windows 原生环境中谨慎处理。
- 第 4 步：先抽样修改 1 到 2 个文件，并确认显示正常。
- 第 5 步：确认无乱码、无结构破坏后，再扩大修改范围。
- 第 6 步：如果任何一步出现疑似乱码，立即停止继续写入，先排查来源。
