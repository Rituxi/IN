# 智能服务迁移说明书

## 1. 文档用途

这份说明书用于把当前小程序里的智能能力单独拆成一个网站应用或后端服务，方便后续：

- 重做智能 OCR 网站应用
- 重做智能小结网站应用
- 给小程序继续调用
- 后面更换域名时只改少量配置

这份文档分成两部分：

1. 业务需求说明
2. 当前小程序真实对接格式说明

---

## 2. 当前项目里的智能服务现状

当前小程序已经接了 4 类智能相关能力：

1. OCR 识别服务
2. 智能小结服务
3. 用户额度查询服务
4. 兑换码服务

当前服务地址来源：

- 主配置在 `app.js` 的 `OCR_SERVERS`
- OCR 接口路径：`/api/analyze/image-base64`
- 智能小结接口路径：`/api/summary/text`
- 用户额度接口路径：`/api/user/quota`
- 兑换码接口路径：`/api/user/redeem`

当前小程序的设计思路是：

- 小程序只管上传数据和接收结果
- 智能识别、智能小结、额度限制、兑换码都由网站后端统一处理
- 小程序本地只负责保存记录和展示

所以后面如果迁移新服务，最好继续保持这个模式。

---

## 3. 本次要做的网站应用需求

根据现有需求，网站应用建议包含下面 4 个模块。

### 3.1 模块一：智能 OCR

用途：

- 网站端直接上传报告单图片进行识别
- 小程序也能把图片传给网站接口做识别
- 返回结构化的检验指标结果

要求：

- 支持单张报告单图片上传
- 支持识别医院、日期、医生、检查项、结果、单位、参考范围
- 支持识别后返回给小程序使用
- 支持额度控制
- 支持失败错误提示

扩展要求：

- 后续支持 Excel 导入历史指标
- Excel 导入后也要转换成统一格式

### 3.2 模块二：智能小结

用途：

- 接收小程序或网站传来的指标数据
- 调用 AI 模型生成指标说明/智能小结

要求：

- 输入是结构化指标数据，不是图片
- 输出是一段可直接展示给用户的总结文字
- 支持额度控制
- 支持返回剩余次数

### 3.3 模块三：用户等级与配额

用途：

- 根据用户 ID 区分用户身份和可用次数

当前需求分 3 档：

- `care`
- `care_plus`
- `king`

规则说明：

- `care`：普通用户
- `care_plus`：高级用户
- `king`：无限畅享用户

支持能力：

- 查询当前用户等级
- 查询 OCR 剩余次数
- 查询智能小结剩余次数
- 通过兑换码升级用户等级
- 后台可手动加次数、升级、降级、取消升级

### 3.4 模块四：后台管理

后台建议至少有这 4 页：

1. 首页统计
2. 使用记录
3. 用户列表
4. 兑换码管理

#### 首页统计

显示：

- 总调用次数
- 今日调用次数
- 本月调用次数
- 总用户数

#### 使用记录

显示：

- 用户 ID
- 用户 IP
- IP 属地（省 + 市）
- 使用功能
- 本月第几次
- 累计使用次数
- 使用时间
- 状态

#### 用户列表

显示：

- 用户 ID
- IP
- OCR 本月已使用 / 额度
- 智能小结本月已使用 / 额度
- 累计使用次数
- 备注信息
- 初次使用时间
- 当前等级状态

支持操作：

- 加次数
- 升级 care_plus
- 升级 king
- 取消 care_plus
- 取消 king
- 按等级筛选
- 按关键词搜索
- 显示用户数量
- 刷新列表

#### 兑换码管理

支持：

- 生成 care_plus 兑换码
- 生成 king 兑换码
- 刷新列表
- 删除兑换码
- 复制兑换码
- 禁用兑换码
- 隐藏已删除或已失效的兑换码

## 5. 当前域名和迁移方式

### 5.1 当前主域名配置

当前小程序通过下面两个域名轮流调用：

```javascript
[
  "https://zbbj2.zeabur.app",
  "https://zbbj1.zeabur.app"
]
```

### 5.2 后续迁移时优先修改的位置

优先修改：

`app.js` 中的 `OCR_SERVERS`

如果只换主域名，并且接口路径不变，一般改这里就够了。

### 5.3 建议继续保留的接口路径

为了让迁移成本最低，建议新服务继续保留这些路径：

```text
/api/analyze/image-base64
/api/summary/text
/api/user/quota
/api/user/redeem
```

这样小程序代码可以少改很多。

### 5.4 如果要彻底替换服务，建议同步检查这几个位置

1. `app.js` 的 `OCR_SERVERS`
2. `packageSub/diary-add/diary-add.js` 里的 OCR 备用地址
3. `packageSub/diary-add/diary-add.js` 里的智能小结备用地址

如果新服务已经完全可用，而且 `app.js` 能正常提供服务地址，通常实际运行时主要还是走 `app.js` 里的配置。

## 6. 用户基础身份格式

当前小程序依赖两个基础身份字段：

- `userId`
- `userLevel`

### 6.1 userId 格式

当前小程序自动生成设备级唯一 ID，示例：

```json
{
  "userId": "dev_m8v4h2abc123de456789"
}
```

格式规则：

```text
dev_ + 时间戳36进制 + 随机串
```

后端应把 `userId` 当成唯一用户标识。

### 6.2 userLevel 枚举

当前小程序使用这 3 个值：

```json
["care", "care_plus", "king"]
```

说明：

- `care`：普通用户
- `care_plus`：高级用户
- `king`：无限用户

## 7. OCR 接口格式

### 7.1 接口地址

```text
POST /api/analyze/image-base64
```

### 7.2 小程序当前请求格式

当前小程序发给 OCR 的请求体示例：

```json
{
  "base64": "data_removed_base64_string",
  "mimeType": "image/jpeg",
  "userId": "dev_m8v4h2abc123de456789",
  "nickname": "匿名"
}
```

字段说明：

- `base64`：图片 Base64 内容，不带额外业务结构
- `mimeType`：当前固定常用 `image/jpeg`
- `userId`：用户唯一 ID
- `nickname`：当前可传匿名，后端可选用

### 7.3 小程序当前成功返回格式

当前小程序对 OCR 成功结果的期望，不要求一定有 `success: true`，它更看重返回对象里有没有 `error` 字段。

建议成功返回示例：

```json
{
  "title": "门诊复查记录",
  "date": "2026-03-16",
  "hospital": "浙江大学医学院附属某医院",
  "doctor": "张医生",
  "notes": "尿常规部分字体较浅，已按可见内容识别",
  "items": [
    {
      "id": "creatinine",
      "name": "肌酐",
      "value": "86",
      "unit": "umol/L",
      "range": "44-133"
    },
    {
      "id": "urine_pro_quant",
      "name": "尿蛋白定量",
      "value": "0.12",
      "unit": "g/L",
      "range": "<0.15"
    }
  ]
}
```

### 7.4 OCR 返回字段说明

推荐字段：

- `title`
- `date`
- `hospital`
- `doctor`
- `notes`
- `items`

其中 `items` 每项格式：

```json
{
  "id": "creatinine",
  "name": "肌酐",
  "value": "86",
  "unit": "umol/L",
  "range": "44-133"
}
```

注意：

- `id` 可以有，也可以没有
- 小程序会优先用 `name` 去匹配自己的指标库
- `value` 当前建议返回字符串，兼容性更稳

### 7.5 OCR 失败返回格式

当前小程序把有 `error` 的响应视为失败。

建议失败示例：

```json
{
  "error": "QUOTA_EXCEEDED",
  "message": "本周免费额度已用完"
}
```

建议错误码至少支持：

```json
[
  "QUOTA_EXCEEDED",
  "IMAGE_ANALYZE_FAILED",
  "INVALID_IMAGE",
  "UNSUPPORTED_FILE",
  "RATE_LIMIT",
  "SERVER_ERROR"
]
```

## 8. 智能小结接口格式

### 8.1 接口地址

```text
POST /api/summary/text
```

### 8.2 小程序当前请求格式

当前小程序请求体示例：

```json
{
  "userId": "dev_m8v4h2abc123de456789",
  "examData": {
    "date": "2026-03-16",
    "items": [
      {
        "name": "肌酐",
        "value": "86",
        "unit": "umol/L",
        "range": "44-133"
      },
      {
        "name": "尿蛋白定量",
        "value": "0.12",
        "unit": "g/L",
        "range": "<0.15"
      }
    ]
  },
  "promptSlot": "slot1",
  "nickname": "用户",
  "userLevel": "normal",
  "model": "gemini-2-flash"
}
```

字段说明：

- `userId`：用户唯一 ID
- `examData.date`：检查日期，字符串格式
- `examData.items`：结构化指标列表
- `promptSlot`：提示词槽位，当前小程序写死为 `slot1`
- `nickname`：用户昵称
- `userLevel`：当前小程序这里写的是 `normal`，新服务可以兼容处理
- `model`：模型名称，当前写的是 `gemini-2-flash`

### 8.3 智能小结成功返回格式

这个接口当前小程序要求返回 `success: true`。

示例：

```json
{
  "success": true,
  "summary": "本次检查中肌酐处于正常范围，尿蛋白定量接近上限，建议继续观察并按医嘱复查。",
  "quota": {
    "remaining": 24,
    "used": 1,
    "limit": 25,
    "resetAt": "2026-04-01 00:00:00"
  }
}
```

当前小程序最核心会读取：

- `success`
- `summary`
- `quota.remaining`（可选）

### 8.4 智能小结失败返回格式

示例：

```json
{
  "success": false,
  "error": "QUOTA_EXCEEDED",
  "message": "本月次数已用完"
}
```

建议错误码：

```json
[
  "QUOTA_EXCEEDED",
  "RATE_LIMIT",
  "SUMMARY_FAILED",
  "IMAGE_LIMIT_EXCEEDED",
  "SERVER_ERROR"
]
```

## 9. 用户额度查询接口格式

### 9.1 接口地址

```text
GET /api/user/quota
```

### 9.2 小程序当前请求格式

请求参数：

```json
{
  "userId": "dev_m8v4h2abc123de456789"
}
```

### 9.3 小程序当前成功返回格式

当前小程序要求：

- HTTP 200
- `res.data.success === true`

数据建议示例：

```json
{
  "success": true,
  "data": {
    "userId": "dev_m8v4h2abc123de456789",
    "isPro": true,
    "isUnlimited": false,
    "extraQuota": 25,
    "ocrUsed": 12,
    "ocrLimit": 50,
    "summaryUsed": 8,
    "summaryLimit": 50,
    "resetAt": "2026-04-01 00:00:00"
  }
}
```

### 9.4 小程序如何判断等级

当前逻辑：

- `isUnlimited === true`，则判定为 `king`
- 否则如果 `isPro === true` 或 `extraQuota > 0`，判定为 `care_plus`
- 否则判定为 `care`

所以如果你想兼容当前小程序，后端最好至少返回：

```json
{
  "isUnlimited": false,
  "isPro": true,
  "extraQuota": 25
}
```

## 12. items 指标项格式

这是 OCR、智能小结、小程序展示三方都要统一的核心结构。

单条 `item` 推荐统一格式：

```json
{
  "id": "creatinine",
  "name": "肌酐",
  "value": "86",
  "unit": "umol/L",
  "range": "44-133",
  "categoryName": "肾功能",
  "isQualitative": false
}
```

字段说明：

- `id`：指标标准 ID，可选但强烈建议有
- `name`：指标名称，必须有
- `value`：检测结果
- `unit`：单位
- `range`：参考范围
- `categoryName`：分类名，可选
- `isQualitative`：是否为定性指标，可选

如果后端暂时给不了 `categoryName` 和 `isQualitative`，小程序也能继续工作。

## 16. 建议的后端实现原则

为了让以后迁移最省事，建议后端遵循下面 6 条。

1. 尽量保留当前接口路径不变
2. 尽量保留当前请求字段名不变
3. OCR 返回继续用 `items` 数组
4. 智能小结继续返回 `success + summary + quota`
5. 用户等级继续兼容 `care / care_plus / king`
6. 所有错误尽量返回可读的 `message`

这样小程序后续改动会很小。

## 17. 最省事的迁移方案

如果后面你重做了新的智能服务，最省事的做法是：

1. 新服务继续兼容当前 4 个接口路径
2. 新服务继续兼容当前请求体结构
3. 新服务继续兼容当前返回字段
4. 小程序只改 `app.js` 中的域名配置

这样通常可以做到：

- 旧小程序页面不用大改
- OCR 页面不用重写
- 智能小结页面不用重写
- 兑换码和额度逻辑也能继续沿用

---

## 18. 交付给开发时可以直接说的话

可以直接把下面这段发给开发：

> 需要做一个带后台的网站应用，提供 4 个接口：OCR 识别、智能小结、用户额度查询、兑换码兑换。接口路径尽量保持现有小程序兼容：`/api/analyze/image-base64`、`/api/summary/text`、`/api/user/quota`、`/api/user/redeem`。请求和返回格式请严格按《智能服务迁移说明书》执行，这样后续小程序只需要替换域名，不需要大改代码。

---

## 19. 结论

你现在这套小程序，已经把“前端怎么接智能服务”这层规范基本定下来了。

后面最合适的做法不是推翻重来，而是：

- 把智能服务独立出来
- 后端按当前格式兼容
- 小程序继续复用原有页面
- 真正迁移时主要改域名

这样成本最低，最稳，也最方便你以后继续扩展。
